# 하롱 여행 맞춤 추천 시스템 점검 및 구현 계획

## 1. 결론

한 번의 범주 선택만으로는 고객 맞춤 추천을 만들 수 없다. 현재 추천기는 `가족`, `인원` 같은 단어가 있으면 트리플·커넥팅 객실을 찾고 최저 성인 요금 순으로 3개를 반환한다. 출발일, 일정, 성인·아동 구성, 아동 나이, 객실 수, 총예산, 이동 조건을 누적하지 않으므로 결과는 상품 탐색용으로만 사용해야 한다.

권장 방식은 한 번에 질문 하나를 보여주고 답을 `recommendation_context`에 누적한 뒤, 필수 조건이 모였을 때만 후보를 계산하는 방식이다.

추가로 확인된 문제:

- `cruise_info`, `cruise_rate_card`, `cruise_location` 등이 외래키 대신 크루즈 이름 문자열로 연결된다.
- `valid_year`, `valid_from`, `valid_to`, 프로모션 날짜가 `text`다.
- 휴일 할증 금액과 옵션 금액 일부도 `text`다.
- `cruise_tour_options`에는 기본키가 없다.
- `cruise_promotion_rate.promotion_id`는 UUID 외래키가 아니라 `text`다.
- 현재 상품 상세 화면은 공개 클라이언트에서 여러 테이블을 `select('*')`로 전체 조회한다. 공개에 필요한 열만 제공하는 읽기 모델로 줄여야 한다.
- `price_adult`가 1인 가격인지 객실 기본 가격인지 데이터 사전으로 확정되지 않았다. 확정 전에는 UI에서 `/ 1인`이라고 단정하면 안 된다.
- 당일 크루즈 일부 행은 `max_guests=1`로 들어 있어 객실 수용 인원과 좌석/요금 단위의 의미가 섞였을 가능성이 있다.

## 3. 고객 질문 흐름

질문은 한 화면에 여러 개를 펼치지 않고 하나씩 표시한다. 이전 답변을 수정할 수 있어야 한다.

### 필수 질문

1. 여행 형태
   - 문구: `어떤 하롱 여행을 찾고 계세요?`
   - 선택: `당일 크루즈`, `1박 2일`, `2박 3일`, `아직 모르겠어요`
   - 저장: `schedule_type`

2. 출발일
   - 문구: `언제 출발할 예정인가요? 요금과 가능 상품이 날짜에 따라 달라져요.`
   - 선택: 날짜 입력, `날짜 미정`
   - 저장: `checkin_date`

3. 여행 인원
   - 문구: `누가 함께 여행하나요?`
   - 입력: 성인 수, 아동 수, 유아 수
   - 아동이 있으면 다음 질문으로 각 아동의 여행일 기준 만 나이를 받는다.
   - 저장: `adults`, `child_ages`, `infant_ages`

4. 객실 구성
   - 숙박 상품 문구: `객실은 몇 개가 필요하고, 함께 묵어야 하는 분이 있나요?`
   - 선택: 객실 수, 트리플, 커넥팅, 엑스트라베드, 싱글룸
   - 당일 상품에는 이 질문을 생략한다.
   - 저장: `room_count`, `room_preferences`

5. 예산
   - 문구: `전체 여행 예산은 어느 정도인가요?`
   - 선택: 구간 선택 또는 직접 입력, `예산보다 조건이 중요해요`
   - 저장: `total_budget_vnd`

### 선택 질문

6. 가장 중요한 경험
   - 문구: `무엇을 가장 중요하게 보고 계세요? 최대 2개를 골라주세요.`
   - 선택: `가족 편의`, `허니문/커플`, `전 객실 발코니`, `조용한 분위기`, `시설과 액티비티`, `합리적인 가격`, `VIP 서비스`

7. 이동
   - 문구: `하노이 또는 공항에서 이동도 필요하신가요?`
   - 선택: `하노이 왕복`, `공항 이동`, `하롱 현지 합류`, `나중에 결정`

### 추천 가능 시점

`schedule_type + checkin_date(또는 미정) + 인원 + 숙박 시 객실 구성`이 모이면 1차 후보를 보여준다. 예산과 선호는 순위를 개선하는 조건이다. 아동 나이 규칙이나 날짜 요금이 없는 경우에는 상품 후보만 보여주고 가격은 현지 확인으로 표시한다.

## 4. 권장 DB 구조

기존 테이블을 즉시 삭제하지 말고 `v2` 테이블과 공개 읽기 뷰를 만든 뒤 검증 후 전환한다.

### 핵심 테이블

- `cruises`
  - `id uuid primary key`, `slug unique`, `code unique`, 한/영문 이름, 여행 형태, 설명, 등급, 상태
- `cruise_itineraries`
  - `id`, `cruise_id foreign key`, `schedule_type`, 숙박 일수, 일정 설명
- `cabins`
  - `id`, `cruise_id foreign key`, 객실명, 면적, 침대, 최대 성인/전체 인원, 발코니·커넥팅·엑스트라베드 속성
- `rate_plans`
  - `id`, `cabin_id foreign key`, `itinerary_id foreign key`, `valid_during daterange`, 성인·아동·유아·싱글 요금, `price_basis`, 통화, 활성 여부
- `child_policies`
  - 크루즈/요금제별 최소·최대 나이, 무료/정액/비율 규칙
- `promotions`
  - 예약 기간과 체크인 기간을 `daterange`로 저장하고 요금제에 외래키로 연결
- `cruise_transfers`
  - 크루즈/일정별 출발지, 차량 종류, 왕복 여부, 요금, 활성 여부
- `cruise_tags`
  - `family`, `honeymoon`, `quiet`, `activity`, `luxury`, `value` 등의 추천 태그와 근거

### 공개 읽기 모델

`public_cruise_recommendation_view`에는 고객 추천에 필요한 공개 열만 제공한다. 내부 메모, 원본 URL, 운영 경고 등은 제외한다. 뷰는 `security_invoker = true`를 사용하고 기반 테이블에 RLS 및 명시적 SELECT 정책을 둔다.

### 필수 제약과 인덱스

- 모든 테이블에 기본키와 외래키를 둔다.
- `price >= 0`, `max_guests > 0`, 허용된 `schedule_type`, 허용된 `currency` 체크 제약을 둔다.
- `(cruise_id, schedule_type, active)` 및 `(cabin_id, valid_during)` 조회 인덱스를 둔다.
- 활성 요금만 조회한다면 부분 인덱스 `where is_active = true`를 사용한다.
- 동일 객실·일정·유효기간·요금 유형 중복을 막는 유니크 제약을 둔다.

## 5. 추천 방식

### 1단계: 하드 필터

- 일정 코드 일치
- 출발일이 요금 유효기간에 포함
- 전체 인원 및 객실 배치 가능
- 요청한 객실/커넥팅/엑스트라베드 가능
- 활성 상품과 활성 요금만 포함

### 2단계: 점수화

- 예산 적합도 30점
- 인원·객실 배치 적합도 25점
- 고객 선호 태그 25점
- 추천/데이터 완성도 10점
- 이동 옵션 적합도 10점

점수와 함께 `추천 이유`를 반드시 반환한다. 데이터가 없는 조건은 0점 처리하지 말고 `확인 필요`로 분리한다.

### 3단계: 결과 문구

예시:

> 성인 2명·아동 1명, 1박 2일, 커넥팅 또는 트리플 객실 조건으로 3개를 찾았어요. 아래 금액은 현재 등록 요금 기준이며 아동 나이와 출발일 할증을 반영한 최종 금액은 현지 데스크가 확인합니다.

상품 카드에는 다음만 표시한다.

- 상품명과 일정
- 조건에 맞는 객실
- 계산 근거가 확인된 예상 금액 또는 `요금 확인 필요`
- 추천 이유 2개
- 확인이 필요한 조건

## 6. 구현 순서

### Phase 0 — 데이터 사전 확정

- `price_adult`, 객실 가격, 1인 가격의 의미를 운영 담당자와 확정
- 아동/유아 나이 기준, 당일 크루즈 `max_guests` 의미 확정
- 한국어 표준 상품명과 별칭 매핑 확정

### Phase 1 — 현 데이터 정리

- 상세 없는 요금 상품 9개를 `상세 추가`, `추천 제외`, `별칭 병합`으로 분류
- 중복 요금 8개 해결
- 날짜/금액 문자열을 검증 가능한 타입으로 변환
- 누락된 아동 연령, 객실 면적, 침대 유형, 카테고리 보완

### Phase 2 — v2 스키마와 마이그레이션

- 기존 테이블을 유지한 채 정규화 테이블 생성
- 이름 기반 관계를 UUID 외래키로 변환
- 제약, 인덱스, RLS, 공개 읽기 뷰 추가
- 행 수·최저가·상품별 객실 수 대조 테스트

### Phase 3 — 추천 엔진

- 고객 답변 스키마와 서버 측 검증 추가
- 하드 필터와 점수화 로직 분리
- 추천 이유와 확인 필요 항목 생성
- 추천 가능한 후보가 없으면 조건 완화 질문 또는 상담 이관

### Phase 4 — 단계형 UI

- 한 번에 질문 하나만 표시
- 이전 답변 요약과 수정 기능 제공
- 후보 수에 따라 다음 질문을 선택
- 필수 조건 충족 후에만 추천 실행

### Phase 5 — 평가와 출시

- 성인 커플, 아동 가족, 3세대 가족, 허니문, 당일 여행, 날짜 미정 등 대표 시나리오 평가
- 잘못된 가격 단정 0건, 수용 불가능 객실 추천 0건을 출시 기준으로 설정
- 운영 담당자 승인 후 기존 추천기를 v2로 전환

## 7. 우선 추천

첫 출시 범위는 `상세정보가 있는 17개 상품`, `2026~2027`, `DAY/1N2D/2N3D`, `성인 중심 상품 후보 추천`으로 제한한다. 가족 문의는 아동 나이와 객실 배치까지 받되, 아동 규칙이 없는 요금은 자동 총액 대신 상담 확인으로 표시한다. 실시간 재고 데이터가 없으므로 어떤 경우에도 예약 가능 여부를 확정하지 않는다.
